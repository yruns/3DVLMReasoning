# v10 Evaluation: LLM Object Enrichment Integration

**Date**: 2026-03-30
**Branch**: `feat/stage2-evidence-seeking-v2`
**Evaluation**: OpenEQA ScanNet, 100 cases (20 scenes x 5 questions), scored by Gemini 2.5 Pro
**Key Change**: Integrated `enriched_objects.json` (Gemini-generated object metadata) into Stage 1 pipeline

---

## 1. What Changed in v10

### 1.1 Object Enrichment Data (Phase A)

Each of the 89 OpenEQA ScanNet scenes received an `enriched_objects.json` file generated by Gemini 2.5 Pro vision. For every detected object, Gemini analyzed annotated crops (red bbox, top-3 visibility frames) and produced:

| Field | Description | Example |
|-------|-------------|---------|
| `category` | LLM-verified category label (replaces noisy detection label) | "microwave" → "countertop microwave oven" |
| `description` | Concise visual description (appearance, material, condition) | "A white Sanyo brand countertop microwave..." |
| `location` | Position relative to walls, landmarks | "On the kitchen counter, against the back wall" |
| `nearby_objects` | Directly observed neighboring objects | ["electric stove", "kitchen counter", "wall cabinet"] |
| `color` | Primary color(s) | "white" |
| `usability` | Functional affordances | "Used for heating and cooking food quickly" |

**Production stats:**
- 89/89 scenes processed, 5,097/5,097 objects enriched (100%), 0 failures
- Script: `src/scripts/enrich_objects.py` (~614 lines)

### 1.2 Pipeline Integration (Phase B)

Single code change: `src/query_scene/keyframe_selector.py` (+41 lines)

New `_load_enrichment()` method populates existing `SceneObject` fields:
- `SceneObject.category` ← `enrichment["category"]` (overrides noisy detection label)
- `SceneObject.object_tag` ← synced with category
- `SceneObject.summary` ← `enrichment["description"]`
- `SceneObject.affordance_category` ← `enrichment["usability"]`
- `SceneObject.co_objects` ← `enrichment["nearby_objects"]`
- `SceneObject.affordances` ← full enrichment dict

These fields flow downstream via `stage1_adapters.py:build_object_context()` → `Stage2EvidenceBundle.object_context` → Stage 2's `retrieve_object_context()` tool.

**No silent fallbacks:** Missing `enriched_objects.json` raises `FileNotFoundError`.

## 3. Failure Analysis (36 Low-Score Cases)

All 36 cases with E2E score ≤ 2 were analyzed by 6 parallel reviewer agents. Each case was examined against 10+ raw scene images, stage1/stage2 execution logs, and enriched_objects.json.

### 3.1 Root Cause Distribution

| Root Cause | Count | % of 36 | Description |
|------------|:-----:|:-------:|-------------|
| RETRIEVAL_FAILURE | 22 | 61% | Wrong keyframes, wrong proxy term, target area not covered |
| REASONING_ERROR | 11 | 31% | VLM misidentified objects, spatial relations, colors |
| VISIBILITY_ISSUE | 6 | 17% | Dark lighting, small objects, low resolution |
| AMBIGUOUS_GT | 5 | 14% | Ground truth is debatable or evaluator misjudged |
| LABELING_ERROR | 2 | 6% | Ground truth answer appears factually incorrect |

> Some cases have multiple root causes, so counts sum to > 36.

### 3.2 Top 5 Systemic Issues

#### Issue 1: Tool Callbacks Not Configured (~20 cases affected)

`request_more_views` and `request_crops` return "callback is not configured" in the Stage 2 run. This **completely disables** the evidence-seeking behavior that is Stage 2's core value proposition.

Evidence:
- Case 6 (flower pot color): E2E mode with tools enabled → correctly identified "light brown/tan" via crop. Stage 2 without tools → "insufficient_evidence"
- Cases 3, 4, 5, 9, 10, 12, 14, 15, 16, 18, 19, 20, 23, 25, 26, 28, 35: all attempted tools but failed

**Fix:** Configure tool callbacks in evaluation run. This is a configuration issue, not a code issue.

#### Issue 2: Single-Keyframe Fragility (~18 cases affected)

18 of 36 low-score cases received only 1 keyframe. Spatial understanding, holistic judgment, and fine-detail questions all require multiple viewpoints.

Examples:
- Case 14 ("Is the room organized?"): 1 corner view → "insufficient evidence to judge"
- Case 15 ("What's the room number?"): 1 frame showing cable jack, not door sign
- Case 28 ("Can I throw more trash?"): 1 blurry frame, trash bin barely visible

**Fix:** Enforce minimum keyframe count = 3 for all queries, ≥ 4 for spatial questions.

#### Issue 3: Query Parser Proxy Term Errors (~12 cases affected)

Stage 1 query parser maps queries to incorrect proxy objects, causing retrieval of irrelevant frames:

| Query | Parsed As | Should Be |
|-------|-----------|-----------|
| "red object next to sofa" | target=armchair | color+proximity search |
| "object in middle of floor" | target=bag | floor-level global search |
| "What is behind the sofa?" | target=wall | open-ended spatial search |
| "room number" | target=number (→cable jack) | door/sign composite concept |
| "patio chairs" | target=Queen Anne dining chair | outdoor/window-facing views |
| "What is on top of calculator" | target=Backpack | desk surface search |

**Fix:** Open-ended queries ("What is behind/near X?") should not pre-resolve a target term. Color-described targets should use color-based filtering.

#### Issue 4: Small Object Detection Gap (~8 cases affected)

GSA/RAM detection pipeline misses critical small objects:

| Missing Object | Scene(s) | Impact |
|----------------|----------|--------|
| toothbrush | 013, 102 | Cannot retrieve or answer |
| trash bin/can | 013, 100 | Cannot retrieve |
| hat | 101 | Completely unanswerable |
| calculator | 109 | Spatial relations broken |
| deadbolt | 046 | Fine-detail question fails |
| dryer (vs washer) | 031 | Object confusion |

**Fix:** Expand GSA/RAM detection vocabulary. Consider using enriched `nearby_objects` as secondary search terms when primary detection fails.

#### Issue 5: Keyframe Spatial Diversity (~5 cases affected)

When many instances of the target object exist, keyframes cluster in one region:

- Case 30: 35 office chairs detected, all 3 keyframes from first 10% of trajectory → only black chairs shown, red chairs missed
- Case 33: "between X and Y" query only anchored on X, retrieved wrong corner

**Fix:** High-count targets need spatial-cluster sampling. Bi-anchor queries need frames showing both landmarks.

### 3.3 GT/Evaluator Issues Found

| Case | Issue | Detail |
|------|-------|--------|
| 2 | **Evaluator error** | Pred "A floor lamp" = GT "lamp" → scored 1 (should be 4-5) |
| 9 | **GT error** | GT "vacuum cleaner" but red object is clearly a fire extinguisher |
| 36 | **GT error** | GT "coffee machine" but red object is a KitchenAid stand mixer (coffee maker is black) |
| 8 | **Ambiguous GT** | "green" sofa appears gray/olive under dim lighting; both answers defensible |
| 25 | **Unanswerable** | "What color is my hat?" — first-person object behind camera, not in any frame |
| 35 | **Beyond capability** | "Where can you see the lamp?" GT "in the mirror" — requires mirror-reflection reasoning |
| 19-20 | **Beyond capability** | Patio objects outside scan boundary, only visible through windows |

## 4. Improvement Priority Matrix

| Priority | Fix | Expected Impact | Complexity | Affected Cases |
|:--------:|-----|:---------------:|:----------:|:--------------:|
| **P0** | Enable tool callbacks (request_more_views, request_crops) | ~15-20 cases improved | Low (config) | 3,4,5,6,9,10,12,14,15,16,18,19,20,23,25,26,28,35 |
| **P0** | Enforce min 3 keyframes per query | ~12 cases improved | Low (code) | 3,5,6,10,12,14,15,16,18,21,22,28 |
| **P1** | Open-ended queries: don't pre-resolve target term | ~8 cases improved | Medium | 9,12,20,24,29,30 |
| **P1** | Spatial diversity: cluster-based keyframe sampling | ~5 cases improved | Medium | 14,30,33 |
| **P2** | Expand small-object detection vocabulary | ~8 cases improved | High | 4,5,7,23,25,26,32 |
| **P2** | Use enrichment color field for color-described queries | ~4 cases improved | Medium | 8,9,18,21 |
| **P3** | Bi-anchor retrieval for "between X and Y" queries | ~3 cases improved | High | 27,33 |
| **P3** | Outdoor/window-facing retrieval for patio queries | ~2 cases improved | High | 19,20 |

**Conservative estimate: P0 fixes alone (callbacks + min keyframes) could improve E2E mean from 3.21 to ~3.5+**
